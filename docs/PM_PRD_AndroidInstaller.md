# 产品需求说明书 (PRD)

## 1. 项目背景与目标
针对测试及研发过程中频繁需要给移动设备安装 `.apk` 文件的诉求，为取代繁琐的手工命令行 (`adb install`) 流程，现需要开发一款桌面端可视化工具应用。
**目标：** 提供一个易用、高效的可视化操作界面，降低安装包部署的门槛，使得任何成员均可直观地一键完成安卓设备上的安装和调试工作。

## 2. 目标用户
- **QA/测试工程师**：能够通过选择本地打的包，快速安装到指定测试机上测试。
- **策划/运营/非技术人员**：无需了解复杂的 ADB 命令或终端配置即可实现本地安装包体验。
- **开发人员**：提供实时的设备状态响应及底层日志打印，便于排障。

## 3. 核心功能需求
### 3.1. 跨平台多 Tab 架构
- **描述**：应用主界面须支持选项卡（Tab）切换形式。
- **要求**：第一期主要实现第一顺位的「Android 工具」Tab；并预留比如「iOS 工具」、「PC 工具」等结构，满足未来其他脚本的无限拓展。

### 3.2. 设备管理与选择 (Android版块)
- **描述**：识别当前 PC 连接的所有在线的 Android 设备。
- **要求**：
  - 提供设备下拉选择框（Dropdown List）。
  - 支持手动刷新或每次点击下拉单时自动识别已连接电脑并且授权的 ADB 设备。
  - 界面直观显示当前选中的设备串号。

### 3.3. APK 文件定位与选择
- **描述**：在本地文件系统中定位安装包所在的目录，并展示安装包列表。
- **要求**：
  - 提供“选择目录”按钮，打开系统资源管理器选择路径。
  - 选择目标目录后，在界面通过列表 (List/Dropdown) 展现当前目录及其子目录下后缀为 `.apk` 的本地文件。

### 3.4. 安装控制与过程反馈
- **描述**：开始安装流程，并在界面上反馈即时状态。
- **要求**：
  - 包含一个清晰可见的“安装”按钮。
  - 安装过程中界面必须有明确的进度反馈（如进度条或状态文字，如“正在传输文件…”、“正在安装…”），防止用户误以为软件卡死。
  - **安装日志模块**：底部必须包含一个大的多行文本区域，实时打印控制台中执行 ADB 时的终端返回结果日志（包括失败的报错提示），帮助用户分析安装失败的原因。

## 4. UI 交互体验设计规范 (UX)
- 要求交互简洁直接（从上至下：选设备 -> 选文件 -> 点安装 -> 看进度/日志）。
- 在安装过程中需要对“安装按钮”进行禁用（Disable）处理，避免由于多次点击引起的重复执行问题。
